# Network (네트워크)

## HTTP (HyperText Transfer Protocol)

**한줄설명:** 웹 브라우저와 서버가 문서, 이미지, 데이터 등을 주고받는 통신 규약

**상세설명:**
- 특징:
  - **요청-응답 방식**: 클라이언트가 요청하면 서버가 응답
  - **상태 비저장 (Stateless)**: 각 요청은 독립적
  - **텍스트 기반**: 읽기 쉬운 형식
  - **포트**: 기본값 80
- 메서드:
  - GET: 데이터 조회 (URL에 파라미터 표시)
  - POST: 데이터 생성 (바디에 파라미터)
  - PUT: 데이터 수정
  - DELETE: 데이터 삭제
- 상태 코드:
  - 1xx: 정보
  - 2xx: 성공
  - 3xx: 리다이렉트
  - 4xx: 클라이언트 오류
  - 5xx: 서버 오류

**리뷰:** 웹의 기초가 되는 프로토콜. 깊은 이해 필수

---

## HTTPS (HTTP Secure)

**한줄설명:** HTTP 통신에 SSL/TLS 보안 계층을 추가한 버전

**상세설명:**
- 특징:
  - **암호화**: 통신 내용을 암호화하여 도청 방지
  - **무결성**: 데이터 변조 방지
  - **인증**: 서버 신원 확인
  - **포트**: 기본값 443
- SSL/TLS 핸드셰이크:
  1. 클라이언트가 서버에 연결 요청
  2. 서버가 인증서 전송
  3. 클라이언트가 인증서 검증
  4. 공개키로 세션키 암호화하여 전송
  5. 이후 통신은 세션키로 암호화
- 현재: HTTPS는 필수. HTTP는 거의 사용하지 않음

**리뷰:** 보안의 기본. 모든 공개 서비스는 HTTPS 필수

---

## SSH (Secure Shell)

**한줄설명:** 안전하게(원격으로) 다른 컴퓨터에 접속할 수 있도록 해주는 네트워크 프로토콜

**상세설명:**
- 특징:
  - 원격 접속 후 명령어 입력 가능
  - 암호화된 안전한 통신
  - 비밀번호 또는 공개키-비공개키 인증
- 사용:
  ```bash
  ssh user@hostname -p 22
  ```
- 인증 방식:
  - **비밀번호 인증**: 취약함 (Brute Force 공격 가능)
  - **공개키 인증** (Type 2): 안전함 (회사 보안 표준)
- 포트: 기본값 22

**리뷰:** 서버 관리의 필수 기술. 공개키 인증 방식 사용 권장

---

## 프로토콜 (Protocol)

**한줄설명:** 상호 간의 약속, 전송 규약, 통신 규약

**상세설명:**
- 통신 참여자들이 지켜야 할 규칙
- 예:
  - HTTP: 웹 통신
  - FTP: 파일 전송
  - SMTP: 메일 송신
  - POP3: 메일 수신
  - SSH: 원격 접속
  - DNS: 도메인 → IP 변환
- 각 프로토콜은 특정 계층에서 작동 (TCP/IP 모델)

**리뷰:** 네트워크 통신의 기초 개념

---

## 서버 (Server)

**한줄설명:** 요청을 받았을 때 응답(response)해주는 기기 또는 소프트웨어

**상세설명:**
- 종류:
  - **웹 서버**: HTTP 요청에 웹 페이지 응답
  - **애플리케이션 서버**: 비즈니스 로직 처리
  - **데이터베이스 서버**: 데이터 저장/조회
  - **DNS 서버**: 도메인 → IP 변환
  - **메일 서버**: 메일 송수신
- 특징: 24시간 켜져 있어야 함 (가용성)

**리뷰:** 클라이언트-서버 아키텍처의 핵심 요소

---

## 클라이언트 (Client)

**한줄설명:** 요청(request)하는 쪽 (사용자나 프로그램)

**상세설명:**
- 서버에 요청을 보내고 응답을 받음
- 예:
  - 웹 브라우저: 웹 서버에 요청
  - 모바일 앱: 백엔드 서버에 API 요청
  - 데스크톱 프로그램: 서버에 데이터 요청
- 특징: 사용자의 명령에 따라 작동

**리뷰:** 클라이언트-서버 아키텍처 이해의 기본

---

## 클라우드 (Cloud Computing)

**한줄설명:** 인터넷을 통해 원격 서버에 데이터를 저장하고 소프트웨어를 실행시키는 방식

**상세설명:**
- 특징:
  - **중앙 집중형**: 모든 리소스가 클라우드 데이터 센터에 위치
  - **탄력성**: 필요에 따라 리소스 확장/축소
  - **비용 효율**: 사용한 만큼 지불 (Pay-as-you-go)
  - **관리 용이**: 인프라 관리 불필요
- 서비스 모델:
  1. **IaaS (Infrastructure as a Service)**:
     - AWS EC2, Google Compute Engine
     - 기본 컴퓨팅 리소스만 제공
  2. **PaaS (Platform as a Service)**:
     - Heroku, Google App Engine
     - 개발 환경과 배포 인프라 제공
  3. **SaaS (Software as a Service)**:
     - Google Workspace, Microsoft 365
     - 완성된 소프트웨어 서비스 제공
- 주요 제공자: AWS, Google Cloud, Microsoft Azure, Naver Cloud

**리뷰:** 현대 IT의 표준. 어느 모델을 사용하는지 이해 필수

---

## 엣지 컴퓨팅 (Edge Computing)

**한줄설명:** 데이터 생성 지점, 즉 네트워크의 끝단(edge)에서 데이터를 처리하는 방식

**상세설명:**
- 클라우드와의 차이:
  - **클라우드**: 중앙 데이터 센터에서 처리
  - **엣지**: 데이터 생성 지점에서 즉시 처리
- 장점:
  - **낮은 지연시간**: 원격 전송 불필요
  - **대역폭 절감**: 필요한 데이터만 전송
  - **개인정보 보호**: 민감 데이터 로컬 처리
- 사용 사례:
  - 자율주행차
  - 스마트 팩토리
  - IoT 기기

**리뷰:** 5G와 IoT 시대의 중요한 아키텍처 패턴

---

## 온프레미스 (On-Premise)

**한줄설명:** 소프트웨어나 서버를 자신의 시설에 직접 구축하여 운영하는 방식

**상세설명:**
- 특징:
  - **직접 관리**: 모든 시스템을 자신이 관리
  - **높은 초기 비용**: 하드웨어, 소프트웨어 구매
  - **완전한 제어**: 커스터마이징 자유도
  - **유지보수**: 자신의 IT 팀이 담당
- 클라우드와의 비교:
  - 온프레미스: 높은 초기 비용, 낮은 월간 비용
  - 클라우드: 낮은 초기 비용, 높은 월간 비용
- 선택 기준:
  - 데이터 보안이 매우 중요할 때 → 온프레미스
  - 변동성 높고 탄력성 필요 → 클라우드

**리뷰:** 여전히 많은 기업에서 사용, 하이브리드 방식도 증가

## 도메인과 DNS (Domain Name System)

**한줄설명:** 도메인 이름(www.example.com)을 IP 주소(192.168.1.1)로 변환하는 서비스

**상세설명:**
- **도메인**: 웹사이트를 식별하는 이름
  - 예: google.com, github.com
  - 구조: subdomain.domain.extension (www.example.com)
- **DNS**: 도메인 → IP 변환
  - DNS 서버: 도메인과 IP 주소 매핑 데이터 저장
  - 조회 과정:
    1. 로컬 DNS 캐시 확인
    2. 없으면 ISP DNS 서버에 질의
    3. 없으면 공용 DNS (예: Google 8.8.8.8) 질의
    4. 최종적으로 권한 있는 DNS 서버에서 조회
- DNS 레코드:
  - **A**: 도메인 → IPv4 주소
  - **AAAA**: 도메인 → IPv6 주소
  - **CNAME**: 도메인 → 다른 도메인
  - **MX**: 메일 서버 주소

**리뷰:** 웹의 기본 인프라. DNS 문제는 많은 서비스 장애의 원인

---

## DHCP (Dynamic Host Configuration Protocol)

**한줄설명:** 네트워크 연결 시 자동으로 IP 주소를 할당해주는 프로토콜

**상세설명:**
- 기능:
  - IP 주소 자동 할당
  - 게이트웨이, DNS 서버 정보 제공
  - IP 주소 임대 기간 설정
- 과정:
  1. 클라이언트가 DHCP 요청
  2. DHCP 서버가 사용 가능한 IP 제안
  3. 클라이언트가 IP 수락
  4. DHCP 서버가 IP 확정
- 장점:
  - 관리자가 IP를 일일이 할당할 필요 없음
  - 유연한 IP 관리

**리뷰:** 가정용 네트워크에서 기본 기능

## OSI 7계층

**한줄설명:** 국제표준화기구(ISO)가 만든 네트워크 통신을 7단계로 나눈 모델

**상세설명:**
- 목적: 각 계층이 맡는 역할과 기능을 명확히 하여 원활한 통신 지원
- 구조 (하위 → 상위):
  1. **물리 계층 (Physical)**: 전기 신호, 케이블, 하드웨어
  2. **데이터 링크 계층 (Data Link)**: MAC 주소, 프레임, 스위치
  3. **네트워크 계층 (Network)**: IP 주소, 라우팅, 라우터
  4. **전송 계층 (Transport)**: TCP, UDP, 포트
  5. **세션 계층 (Session)**: 연결 설정/유지/종료
  6. **표현 계층 (Presentation)**: 암호화, 압축, 형식 변환
  7. **응용 계층 (Application)**: HTTP, FTP, SMTP, DNS
- 메모리 기법: **물데네전세표응**
- 참고: 실제로는 항상 7계층을 모두 사용하지 않음. TCP/IP 모델이 더 실용적

**리뷰:** 네트워크의 기초 개념이지만, 실제 개발에서는 TCP/IP 모델 중심으로 사용

## TCP/IP 모델

**한줄설명:** 인터넷과 네트워크 통신의 기본 프로토콜 모음

**상세설명:**
- **TCP**: Transmission Control Protocol
  - 신뢰성 있는 연결형 프로토콜
  - 순서 보장, 오류 체크
  - 느리지만 안정적
  - 예: HTTP, HTTPS, FTP, Telnet, SMTP

- **IP**: Internet Protocol
  - 데이터 패킷 전송
  - 주소 지정 (IP 주소)
  - 경로 결정 (라우팅)

- **UDP**: User Datagram Protocol
  - 비연결형, 신뢰성 낮음
  - 빠른 전송
  - 예: DNS, VoIP, 온라인 게임

- 계층 구조:
  ```
  응용 계층: HTTP, FTP, SMTP, DNS
  ↓
  전송 계층: TCP, UDP
  ↓
  인터넷 계층: IP
  ↓
  링크 계층: 이더넷, Wi-Fi
  ```

**리뷰:** 현대 네트워크의 기초. OSI 7계층보다 더 실용적

---

## IP 주소 (IP Address)

**한줄설명:** 네트워크 상에서 각 장치(컴퓨터, 스마트폰 등)를 구분하는 고유한 주소

**상세설명:**
- **IPv4**:
  - 32비트 (4바이트) 주소
  - 표기법: ooo.ooo.ooo.ooo (옥텟)
  - 범위: 0.0.0.0 ~ 255.255.255.255
  - 총 약 43억 개의 주소 (현재 부족)
  - 예: 192.168.1.100

- **IPv6**:
  - 128비트 주소
  - 충분한 주소 공간
  - 점진적으로 도입 중

- **공인 IP (Public IP)**:
  - 전세계에서 유일한 IP
  - 인터넷에서 직접 통신 가능
  - ISP에서 할당

- **사설 IP (Private IP)**:
  - 내부 네트워크에서만 사용
  - 공인 IP로 사용 불가
  - 누구나 자유롭게 사용 가능
  - 범위:
    - 10.0.0.0 ~ 10.255.255.255 (10.0.0.0/8)
    - 172.16.0.0 ~ 172.31.255.255 (172.16.0.0/12)
    - 192.168.0.0 ~ 192.168.255.255 (192.168.0.0/16)
  - 외부 통신 시 NAT를 통해 공인 IP로 변환

- **특수 IP**:
  - 127.0.0.1 (localhost): 자신의 컴퓨터
  - 0.0.0.0: 모든 네트워크 주소
  - 255.255.255.255: 브로드캐스트 주소

**리뷰:** 네트워크의 핵심. 공인/사설 IP 구분 필수

---

## MAC 주소 (Media Access Control)

**한줄설명:** 네트워크 장비(랜카드 등) 고유의 물리적 식별 번호

**상세설명:**
- 길이: 48비트 (6바이트)
- 표기법: XX:XX:XX:XX:XX:XX (16진수)
- 예: 00:1A:2B:3C:4D:5E
- 특징:
  - 같은 네트워크 내에서만 유효
  - 제조사별로 고유 번호 할당
  - IP와 달리 지리적 이동 시 변경 불필요
- 계층: 데이터 링크 계층 (Layer 2)

**리뷰:** IP 주소와는 다른 계층에서 작동하는 주소

---

## Encapsulation (캡슐화)

**한줄설명:** 데이터를 전송할 때 각 계층에서 헤더를 덧붙여 포장하는 과정

**상세설명:**
- 예:
  ```
  응용 계층: Data
  → 전송 계층: TCP Header + Data (Segment)
  → 인터넷 계층: IP Header + Segment (Packet)
  → 링크 계층: Ethernet Header + Packet (Frame)
  ```
- 각 계층이 필요한 정보를 헤더에 추가
- 네트워크 오버헤드 발생

**리뷰:** 네트워크 통신의 기본 메커니즘

---

## Decapsulation (역캡슐화)

**한줄설명:** 수신 측에서 포장된 헤더를 계층별로 벗기며 해석하는 과정

**상세설명:**
- 캡슐화의 역 과정
- 각 계층에서 자신의 헤더를 확인하고 제거
- 최종적으로 원본 데이터 추출

**리뷰:** 통신의 양쪽 프로세스를 이해해야 함

---

## GET 파라미터

**한줄설명:** URL의 ? 뒤에 붙어서 키-값 쌍으로 서버로 전달되는 파라미터

**상세설명:**
- 형식: `http://example.com/?key1=value1&key2=value2`
- 특징:
  - 내용이 URL에 노출됨 (보안 취약)
  - 길이 제한 있음 (브라우저마다 다름, 보통 2000자)
  - 캐싱 가능
  - 북마크 가능
- 예:
  ```
  GET /search?q=javascript&page=1 HTTP/1.1
  ```

**리뷰:** 데이터 조회 시 사용하는 표준 방식

---

## POST 파라미터

**한줄설명:** HTTP 요청의 바디(본문) 안에 포함되는 파라미터

**상세설명:**
- 특징:
  - 내용이 URL에 노출되지 않음 (더 안전)
  - 길이 제한 없음
  - 캐싱 불가
  - 북마크 불가
- 사용 사례:
  - 비밀번호, 개인정보 등 민감한 정보
  - 대량의 데이터
  - 파일 업로드
- 예:
  ```
  POST /login HTTP/1.1
  Content-Type: application/x-www-form-urlencoded

  username=john&password=secret
  ```

**리뷰:** 데이터 생성/수정 시 사용. 비밀번호는 POST로만 전송

---

## HTTP 요청 및 응답 구조

**한줄설명:** HTTP 통신의 기본 구조

**상세설명:**
- **HTTP 요청 (Request)**:
  ```
  GET /index.html HTTP/1.1
  Host: www.example.com
  User-Agent: Mozilla/5.0
  Accept: text/html
  Cookie: sessionId=abc123

  [request body - GET은 보통 없음, POST는 있음]
  ```

- **HTTP 응답 (Response)**:
  ```
  HTTP/1.1 200 OK
  Content-Type: text/html
  Content-Length: 1234
  Set-Cookie: sessionId=xyz789

  <html>
    <body>Hello World</body>
  </html>
  ```

- 구성 요소:
  - **상태 라인**: HTTP 버전 + 상태 코드 + 상태 메시지
  - **헤더**: 메타데이터 (Host, Content-Type, Cookie 등)
  - **바디**: 실제 데이터 (응답은 거의 항상 있음, 요청은 POST 등에만)

**리뷰:** HTTP의 기본 개념

---

## HTTP 상태 코드

**한줄설명:** 서버의 응답 결과를 나타내는 3자리 숫자

**상세설명:**
- **1xx - 정보**: 요청 수신, 계속 진행
  - 100: Continue

- **2xx - 성공**: 요청 성공적으로 처리
  - 200: OK (요청 성공)
  - 201: Created (리소스 생성 성공)
  - 204: No Content (성공하지만 응답 본문 없음)

- **3xx - 리다이렉트**: 추가 조치 필요
  - 301: Moved Permanently (영구 이동)
  - 302: Found (임시 이동)
  - 304: Not Modified (캐시된 버전 사용)

- **4xx - 클라이언트 오류**: 요청에 문제
  - 400: Bad Request (잘못된 요청)
  - 401: Unauthorized (인증 필요)
  - 403: Forbidden (접근 금지)
  - 404: Not Found (리소스 없음)
  - 409: Conflict (충돌)

- **5xx - 서버 오류**: 서버에 문제
  - 500: Internal Server Error (일반적 서버 오류)
  - 503: Service Unavailable (서비스 불가능)

**리뷰:** 면접에서 자주 나오는 주제

  - RFC 7230: HTTP/1.1 Message Syntax and Routing
  - RFC 3986: Uniform Resource Identifier (URI)
  - RFC 2119: Key Words for use in RFCs
- 상태:
  - Draft: 초안
  - Proposed Standard: 제안 표준
  - Internet Standard: 인터넷 표준
- 접근: tools.ietf.org에서 모든 RFC 확인 가능

**리뷰:** 기술 표준의 공식 문서. 상세한 정보 필요 시 참고

## Payload (페이로드)

  POST /api/users HTTP/1.1
  Content-Type: application/json

  {"name": "John", "email": "john@example.com"}  ← 이 부분이 payload

## 3-way Handshake

**한줄설명:** TCP 연결을 설정할 때 사용하는 3단계 통신 절차

**상세설명:**
- 목적: 양쪽이 통신할 준비가 되었는지 확인
- 과정:
  1. **SYN** (클라이언트 → 서버):
     - 클라이언트가 서버에 연결 요청
     - 시퀀스 번호(seq) 포함

  2. **SYN-ACK** (서버 → 클라이언트):
     - 서버가 요청 수락
     - 자신의 시퀀스 번호 전송
     - 클라이언트의 시퀀스 번호에 1을 더한 값 전송 (ACK)

  3. **ACK** (클라이언트 → 서버):
     - 클라이언트가 최종 확인
     - 서버의 시퀀스 번호에 1을 더한 값 전송
     - 연결 확립

- 연결 종료 (4-way Handshake):
  - FIN-ACK-FIN-ACK 순서로 진행

**리뷰:** TCP의 핵심. 면접에서 자주 나오는 주제

---

## 서브넷마스크 (Subnet Mask)

**한줄설명:** IP 주소에서 네트워크 주소와 호스트 주소를 구분하기 위한 방법

**상세설명:**
- 형식: 255.255.255.0 (또는 CIDR 표기: /24)
- 비트 AND 연산으로 네트워크 주소 추출
- 예:
  ```
  IP: 192.168.1.100
  Subnet Mask: 255.255.255.0
  네트워크 주소: 192.168.1.0
  호스트 주소: .100
  ```
- CIDR 표기:
  - /24: 24비트가 네트워크, 8비트가 호스트
  - /32: 모두 네트워크 (호스트 주소 없음)

**리뷰:** IP 주소 계산의 기초

---

## 서브네팅 (Subnetting)

**한줄설명:** 하나의 큰 네트워크(IP 블록)를 여러 개의 작은 네트워크(서브넷)로 나누는 방법

**상세설명:**
- 목적:
  - IP 주소를 효율적으로 사용
  - 네트워크 트래픽 분산
  - 보안성 향상
  - 관리 효율성 증대 (부서별 네트워크 분리)

- 예시:
  ```
  주어진 네트워크 블록: 192.168.1.0/24
  → /24 = 서브넷마스크 255.255.255.0

  호스트 주소: .0 ~ .255 (총 256개)
  실제 사용 가능: .1 ~ .254 (254개)
  - .0: 네트워크 주소
  - .255: 브로드캐스트 주소

  /25로 서브네팅하면:
  - 서브넷 1: 192.168.1.0/25 (호스트 126개)
  - 서브넷 2: 192.168.1.128/25 (호스트 126개)

  /26으로 서브네팅하면:
  - 서브넷 1: 192.168.1.0/26 (호스트 62개)
  - 서브넷 2: 192.168.1.64/26 (호스트 62개)
  - 서브넷 3: 192.168.1.128/26 (호스트 62개)
  - 서브넷 4: 192.168.1.192/26 (호스트 62개)
  ```

**리뷰:** 네트워크 설계의 중요한 기술

---

## NAT (Network Address Translation)

**한줄설명:** 내부 네트워크에서 외부 네트워크로 접속할 때 사설 IP를 공인 IP로 변환해주는 기술

**상세설명:**
- 기능:
  - 여러 대의 기기가 하나의 공인 IP 주소로 인터넷에 접속
  - IP 주소와 포트번호를 변환하여 네트워크 트래픽 주고받음

- 이점:
  - **IP 주소 절약**: 부족한 IPv4 주소를 효율적으로 사용
  - **보안 강화**: 내부 IP 주소가 외부에 노출되지 않음

- 예시:
  ```
  내부 IP: 192.168.1.100:5000 → 외부 IP: 220.100.50.10:8080
  응답: 220.100.50.10:8080 → 192.168.1.100:5000
  ```

**리뷰:** 가정용 공유기에서 기본 사용되는 기술

---

## 특수 IP 주소 (Special IP Address)

**한줄설명:** 일반적인 통신용이 아니라 특수 목적으로 사용하기 위해 예약된 IP 주소

**상세설명:**
- **127.0.0.1 (localhost)**:
  - Loopback 주소
  - 자신의 컴퓨터를 나타냄
  - 네트워크를 거치지 않고 직접 자신에게 돌아옴
  - 개발 및 테스트 시 사용

- **0.0.0.0**:
  - 모든 네트워크 주소를 의미
  - 기본 라우트, 바인드할 모든 인터페이스

- **255.255.255.255**:
  - 브로드캐스트 주소
  - 같은 네트워크의 모든 호스트에 메시지 전송

- **224.0.0.0 ~ 239.255.255.255**:
  - 멀티캐스트 주소
  - 특정 그룹에 메시지 전송

**리뷰:** 개발 시 localhost 사용은 매우 흔함

---

## URI와 URL (Uniform Resource Identifier/Locator)

**한줄설명:** 인터넷 리소스를 식별하고 위치를 지정하는 방식

**상세설명:**
- **URI**: 더 광범위한 개념
  - 리소스의 고유한 식별자
  - URL과 URN을 포함

- **URL**: 위치 기반 식별
  - 예: https://www.example.com/path/to/page
  - 프로토콜://호스트:포트/경로?쿼리#프래그먼트

- **URN**: 이름 기반 식별
  - 예: urn:isbn:0451450523
  - 리소스의 이름으로 식별 (위치 무관)

**리뷰:** 용어의 정확한 의미 이해

---

## TCP (Transmission Control Protocol)

**한줄설명:** 신뢰성 있는 연결형 전송 프로토콜

**상세설명:**
- 특징:
  - **연결 기반**: 3-way Handshake로 연결 설정
  - **신뢰성**: 순서 보장, 오류 재전송
  - **느림**: 안정성을 위해 오버헤드 증가
  - **흐름 제어**: 송신자가 수신자 속도에 맞춤
  - **혼잡 제어**: 네트워크 상태에 따라 전송 속도 조절

- 사용:
  - HTTP/HTTPS
  - FTP
  - Telnet
  - SMTP/POP3

**리뷰:** 신뢰성이 중요한 통신에 사용

---

## UDP (User Datagram Protocol)

**한줄설명:** 빠른 비연결형 전송 프로토콜

**상세설명:**
- 특징:
  - **비연결형**: 연결 설정 없이 바로 전송
  - **비신뢰성**: 손실된 패킷 재전송 없음
  - **빠름**: 최소한의 오버헤드
  - **순서 보장 안 함**: 패킷이 순서대로 도착 안 할 수 있음

- 사용:
  - DNS (도메인 조회)
  - VoIP (음성 통신)
  - 온라인 게임
  - 실시간 스트리밍

**리뷰:** 속도가 중요할 때 사용. 일부 손실은 허용 가능

---

## 로드 밸런싱 (Load Balancing)

**한줄설명:** 네트워크 트래픽을 여러 서버나 자원에 분배하는 기술

**상세설명:**
- 목적:
  - 서버 부하 분산
  - 고가용성 (High Availability)
  - 성능 향상
  - 서버 장애 시 다른 서버로 자동 전환

- 분배 방식:
  - **Round Robin**: 순서대로 분배
  - **Least Connections**: 연결 수가 적은 서버 선택
  - **IP Hash**: 클라이언트 IP 기반으로 일관된 서버 선택
  - **Weighted**: 서버 성능에 따라 가중치 적용
  - **Least Response Time**: 응답 시간이 빠른 서버 선택

- 도구:
  - Nginx
  - HAProxy
  - AWS ELB
  - AWS ALB

**리뷰:** 대규모 서비스 운영의 필수 기술

---

## VPC (Virtual Private Cloud)

**한줄설명:** 클라우드 상에서 격리된 네트워크 공간을 제공하는 서비스

**상세설명:**
- 특징:
  - 사용자가 정의한 네트워크 구조
  - 서브넷, 라우트 테이블, 인터넷 게이트웨이 등 구성 가능
  - IP 주소 범위 설정 가능
  - 다른 VPC와 격리됨 (보안 강화)

**리뷰:** AWS, Google Cloud, Azure 등 클라우드 서비스의 기본 구성 요소

---

## 고가용성 (High Availability)

**한줄설명:** 시스템이 거의 항상 작동 가능한 상태를 유지하는 성질

**상세설명:**
- 가용성: 가동률 (%) = 정상 가동 시간 / 전체 시간
- 목표:
  - 99.9% (Three 9s): 월간 약 43분 다운타임
  - 99.99% (Four 9s): 월간 약 4분 다운타임
  - 99.999% (Five 9s): 월간 약 26초 다운타임

- 달성 방법:
  - 로드 밸런싱
  - 자동 장애 전환
  - 중복 시스템
  - 모니터링과 빠른 복구

**리뷰:** 엔터프라이즈 시스템의 중요한 요구사항
