# Backend (백엔드)

## 백엔드 개발

**한줄설명:** 사용자 요청을 받아서 처리하고 응답하는 서버 측 개발

**상세설명:**
- 백엔드 개발자가 웹 서비스를 만들 때 기본적으로 해야 할 작업:
  1. **라우팅 설계** (맨 앞단):
     - 사용자 인터페이스 역할
     - URL 엔드포인트 정의
     - HTTP 메서드 매핑 (GET, POST, PUT, DELETE 등)

  2. **요청 처리 및 프로세싱** (중간 단):
     - 비즈니스 로직 구현
     - 데이터 검증
     - 권한 확인
     - 데이터 변환

  3. **데이터 관리** (끝단):
     - 데이터베이스에서 데이터 조회
     - 데이터 저장/수정/삭제
     - 트랜잭션 관리

  4. **외부 API 연동**:
     - 타사 서비스 API 호출
     - 결과 처리

**리뷰:** 백엔드의 역할을 이해하는 것이 면접에서 중요

---

## 소켓 통신 (Socket Communication)

**한줄설명:** 실시간 양방향 통신을 가능하게 하는 통신 방식

**상세설명:**
- **TCP 소켓**: 신뢰성 있는 연결형 통신
- **UDP 소켓**: 빠른 비연결형 통신
- **WebSocket**: 웹에서의 실시간 통신
  - HTTP와 다르게 연결 유지
  - 서버에서 클라이언트로 먼저 데이터 전송 가능
  - 채팅, 알림, 실시간 게임 등에 사용
- 용도:
  - 채팅 애플리케이션
  - 실시간 알림
  - 라이브 스트리밍

**리뷰:** 실시간 애플리케이션 개발의 핵심 기술

---

## 웹 서버 (Web Server)

**한줄설명:** HTML, CSS, JS, 이미지 등의 정적 파일을 요청받아서 바로 응답해주는 서버

**상세설명:**
- 특징:
  - 요청을 받아서 내부적으로 다중 프로세스를 통해 빠른 처리 가능
  - 정적 컨텐츠만 제공
  - 가볍고 빠름
- 예시: **Nginx**, Apache, IIS
- 역할:
  - 요청받은 파일을 찾아서 반환
  - 로드 밸런싱
  - SSL/TLS 처리
  - 캐싱

**리뷰:** 프로덕션 환경에서 필수적인 컴포넌트

---

## 웹 애플리케이션 서버 (WAS - Web Application Server)

**한줄설명:** 요청받은 데이터를 처리하는 로직까지 실행할 수 있는 서버

**상세설명:**
- 웹 서버의 기능 + 애플리케이션 실행
- 특징:
  - 동적 컨텐츠 생성 가능
  - 데이터베이스 연동
  - 비즈니스 로직 처리
  - 성능은 웹 서버보다 떨어짐
- 예시: Tomcat, JBoss, Flask, Django, Spring Boot 등
- 사용:
  ```
  요청 → 웹 서버 (Nginx) → WAS (Flask, Tomcat) → DB
  응답 ← 웹 서버 ← WAS ← DB
  ```

**참고:** Flask 자체에 기본 웹 서버 기능도 통합되어 있으나, 프로덕션 환경에서는 Nginx + Gunicorn/uWSGI 조합 사용

**리뷰:** 웹 서버와 WAS의 차이를 명확히 이해할 것

## 마이크로서비스 아키텍처 (Microservices Architecture)

**한줄설명:** 큰 애플리케이션을 작은 독립적인 서비스로 분해하여 개발하는 아키텍처

**상세설명:**
- 특징:
  - **독립적 배포**: 각 서비스를 독립적으로 배포 가능
  - **기술 다양성**: 각 서비스마다 다른 기술 스택 사용 가능
  - **확장성**: 필요한 서비스만 확장 가능
  - **장애 격리**: 한 서비스의 장애가 전체에 영향을 주지 않음
- 단점:
  - 복잡한 배포 구조
  - 네트워크 지연
  - 데이터 일관성 유지 어려움
  - 모니터링 복잡도 증가

**리뷰:** 현대 대규모 시스템의 표준 아키텍처

---

## RESTful API

**한줄설명:** REST 원칙을 따르는 API 설계 방식

**상세설명:**
- REST (Representational State Transfer):
  - **리소스 중심**: 데이터를 리소스로 표현
  - **HTTP 메서드 사용**:
    - GET: 조회
    - POST: 생성
    - PUT: 수정
    - DELETE: 삭제
  - **상태 코드**: 명확한 응답 상태
  - **무상태**: 각 요청은 독립적

- 예시:
  ```
  GET /users - 모든 사용자 조회
  GET /users/1 - ID 1인 사용자 조회
  POST /users - 새로운 사용자 생성
  PUT /users/1 - ID 1인 사용자 수정
  DELETE /users/1 - ID 1인 사용자 삭제
  ```

**리뷰:** 웹 API의 표준. 대부분의 현대 API가 RESTful 원칙 따름

---

## GraphQL

**한줄설명:** 클라이언트가 필요한 데이터를 명확하게 요청할 수 있는 쿼리 언어

**상세설명:**
- REST의 대안
- 특징:
  - **정확한 데이터만**: 필요한 필드만 요청
  - **단일 엔드포인트**: 모든 요청이 하나의 엔드포인트로
  - **강력한 타입 시스템**: 스키마 정의
  - **실시간 구독**: Subscription으로 실시간 업데이트
- 예시:
  ```graphql
  query {
    user(id: 1) {
      name
      email
    }
  }
  ```

**리뷰:** 복잡한 데이터 요구가 많은 프로젝트에서 점점 인기 증가

---

## 인증과 인가 (Authentication & Authorization)

**한줄설명:** 사용자 신원 확인과 권한 부여

**상세설명:**
- **인증 (Authentication)**:
  - 사용자가 누구인지 확인
  - 로그인, 회원가입
  - 방식: 비밀번호, 토큰, MFA 등

- **인가 (Authorization)**:
  - 사용자가 특정 리소스에 접근할 권한이 있는지 확인
  - 역할 기반 접근 제어 (RBAC)
  - 속성 기반 접근 제어 (ABAC)

**리뷰:** 보안의 가장 기본적인 개념

---

## 캐싱 (Caching)

**한줄설명:** 자주 사용되는 데이터나 계산 결과를 빠르게 접근할 수 있는 곳에 저장하는 기법

**상세설명:**
- 종류:
  1. **메모리 캐시**: Redis, Memcached
  2. **HTTP 캐시**: 브라우저, CDN
  3. **데이터베이스 캐시**: 쿼리 결과 캐싱
  4. **애플리케이션 캐시**: 함수 결과 캐싱
- 장점:
  - 응답 속도 향상
  - 서버 부하 감소
  - 데이터베이스 부하 감소
- 단점:
  - 캐시 일관성 유지 어려움
  - 메모리 사용량 증가

**리뷰:** 성능 최적화의 핵심

---

## 로드 밸런싱 (Load Balancing)

**한줄설명:** 네트워크 트래픽을 여러 서버나 자원에 분배하는 기술

**상세설명:**
- 목적:
  - 서버 부하 분산
  - 가용성 증대
  - 성능 향상
- 방식:
  - **Round Robin**: 순서대로 분배
  - **Least Connections**: 연결 수가 적은 서버에 분배
  - **IP Hash**: 클라이언트 IP 기반 분배
  - **Weighted**: 서버 성능에 따른 가중치 적용
- 도구: Nginx, HAProxy, AWS ELB 등

**리뷰:** 대규모 트래픽을 처리하는 필수 기술

---

## 트랜잭션 (Transaction)

**한줄설명:** 하나의 논리적 작업 단위로 모두 성공하거나 모두 실패해야 하는 작업 모음

**상세설명:**
- ACID 원칙:
  1. **원자성 (Atomicity)**: 전부 또는 무(All or Nothing)
  2. **일관성 (Consistency)**: 데이터베이스 무결성 유지
  3. **독립성 (Isolation)**: 동시 실행 시 간섭 없음
  4. **영속성 (Durability)**: 완료 후 영구 저장

- 예시: 계좌 이체
  ```
  계좌 A에서 출금
  계좌 B에 입금
  → 둘 다 성공하거나 둘 다 실패
  ```

**리뷰:** 데이터 무결성의 핵심. 면접 필출 주제

---

## 데이터베이스 정규화 (Normalization)

**한줄설명:** 데이터 중복을 제거하고 데이터 무결성을 유지하도록 테이블을 설계하는 과정

**상세설명:**
- 1정규형 (1NF): 원자적 값만 포함
- 2정규형 (2NF): 부분 함수 종속성 제거
- 3정규형 (3NF): 이행 함수 종속성 제거
- BCNF, 4NF, 5NF 등도 존재

- 장점:
  - 데이터 중복 제거
  - 데이터 무결성 향상
  - 저장 공간 절감

- 단점:
  - 조인이 많아져 조회 성능 저하
  → 필요시 반정규화(Denormalization)

**리뷰:** 데이터베이스 설계의 기본. 과정을 이해해야 함

---

## ERD (Entity Relationship Diagram)

**한줄설명:** 데이터베이스 설계 시 엔티티와 관계를 시각화하는 다이어그램

**상세설명:**
- 구성 요소:
  - **엔티티**: 테이블 (예: User, Product)
  - **속성**: 열 (예: id, name, email)
  - **관계**: 엔티티 간의 연결 (1:1, 1:N, N:N)
- 표기법:
  - Chen 표기법
  - Crow's Foot 표기법 (가장 널리 사용)
  - Barker 표기법

**리뷰:** 데이터베이스 설계 문서화의 표준

---

## 서비스 지향 아키텍처 (SOA - Service Oriented Architecture)

**한줄설명:** 비즈니스 기능을 독립적인 서비스로 구성하는 아키텍처

**상세설명:**
- 마이크로서비스와의 차이:
  - SOA: 엔터프라이즈급, 중앙 ESB(Enterprise Service Bus) 사용
  - 마이크로서비스: 더 작은 단위, 느슨한 결합
- 특징:
  - **재사용성**: 서비스를 다양한 곳에서 재사용
  - **느슨한 결합**: 서비스 간 독립성
  - **상호운용성**: 다양한 기술 조합

**리뷰:** 엔터프라이즈급 시스템의 기본 개념

---

## API 게이트웨이 (API Gateway)

**한줄설명:** 모든 클라이언트 요청을 받아서 적절한 백엔드 서비스로 라우팅하는 중앙 관리 지점

**상세설명:**
- 역할:
  - **요청 라우팅**: 적절한 서비스로 분배
  - **인증/인가**: 모든 요청 검증
  - **레이트 제한**: 트래픽 제한
  - **로깅/모니터링**: 요청 추적
  - **응답 변환**: 일관된 형식으로 반환
- 예시: AWS API Gateway, Kong, Nginx Plus

**리뷰:** 마이크로서비스 아키텍처의 핵심 컴포넌트
